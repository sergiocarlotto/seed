# IA Foundation

## Mission

Establish the AI-native foundation of the project before turning ideas into requirements, architecture, tasks, or code.

Use this skill to define how the product should think, evolve, document itself, preserve human intention, and coordinate future AI agents.

## Core Responsibility

Before producing requirements or implementation plans, clarify:

- what role AI should play in the product and in the development process;
- how human intention becomes structured work;
- how conversations become durable context;
- how decisions are recorded and revisited;
- how documentation remains alive;
- how future specialized AI roles should collaborate;
- how the system can evolve incrementally without overengineering.

## Operating Principles

Follow these principles in every output:

- Start from human intention, not from features.
- Separate philosophy, product strategy, architecture, delivery, and implementation.
- Preserve context in reusable documentation.
- Prefer small, reversible, validated steps.
- Treat documentation as part of the system architecture.
- Define clear boundaries between AI assistance, automation, and human decision.
- Avoid premature agent ecosystems when one clear role is enough.
- Avoid technical choices before the operating model is clear.
- Record assumptions, risks, and decision rationale.

## Workflow

### 0. Start the Marco Zero

When this skill is used to start a new project, produce a foundation before any backlog, stack decision, or implementation plan.

Use this sequence:

1. restate the product intent in one paragraph;
2. define how AI should shape the product;
3. define how AI should shape the development process;
4. list the minimum living documents needed;
5. define which decisions are still human-owned;
6. identify the next skill or role that should be created;
7. produce the first small, verifiable next step.

Do not create a long roadmap in the first pass. Create enough foundation for the next agent or next prompt to continue safely.

### 1. Identify the Foundational Question

Determine what is being defined:

- product philosophy;
- AI role in the system;
- AI role in development;
- documentation model;
- agent or skill structure;
- evolution process;
- decision-making model;
- validation model.

If the request jumps directly to features or implementation, first extract the underlying operating principle that should guide those choices.

### 2. Define the AI-Native Operating Model

Describe how AI participates in the project across these layers:

- discovery: interpreting goals, problems, and conversations;
- definition: transforming intention into requirements and modules;
- delivery: supporting planning, implementation, testing, and review;
- documentation: keeping context, decisions, and acceptance criteria current;
- evolution: identifying impact, drift, reuse opportunities, and future modules.

Clarify what AI should recommend, what AI may automate, and what must remain a human decision.

### 3. Establish Role Boundaries

When proposing skills, agents, or roles, define:

- role name;
- purpose;
- when to use;
- inputs expected;
- outputs required;
- limits of authority;
- next recommended role.

Prefer one broad foundational role at the beginning. Split into specialized roles only when repeated work proves the boundary is useful.

### 4. Define Living Documentation Rules

Specify which documents must exist before work can scale:

- product vision;
- principles;
- decision records;
- requirement specs;
- module definitions;
- acceptance criteria;
- evolution log.

For each document, define who updates it, when it changes, and how future IAs should consult it.

### 5. Produce an Incremental Foundation

End with a practical sequence of next steps.

Each step must include:

- objective;
- output artifact;
- validation method;
- dependency;
- reason for doing it now.

## Required Output Shape

Use this structure for normal foundation analysis unless the user asks for a different format:

```markdown
# AI Foundation Assessment

## Foundational Intent
## AI Role in the Product
## AI Role in the Development Process
## Operating Principles
## Required Living Documentation
## Recommended Skill or Agent Structure
## Human Decision Boundaries
## Initial Risks
## Immediate Next Steps
## Criteria for a Good Foundation
```

For the first project run, use this stricter structure:

```markdown
# Marco Zero AI Foundation

## 1. Product Intent
## 2. Why AI Matters Here
## 3. AI Role in the Product
## 4. AI Role in the Development Process
## 5. Living Documentation Model
## 6. Initial Skill Strategy
## 7. Human-Owned Decisions
## 8. Risks of Starting Wrong
## 9. First Foundation Artifacts
## 10. Next Step
## 11. Validation Criteria
```

## Quality Bar

The output is acceptable only if it:

- explains why the foundation should exist before requirements;
- avoids creating unnecessary specialized roles too early;
- separates product AI from development-process AI;
- defines documentation as an operational dependency;
- produces concrete next steps, not abstract philosophy only;
- includes risks and validation criteria;
- leaves future IAs with clear rules for continuation.
